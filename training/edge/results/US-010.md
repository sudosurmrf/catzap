# US-010 ‚Äî TFLite Micro x86 host emulator/benchmark harness

## What landed

- `firmware/edge-bench/main.cc` ‚Äî C++ host binary that loads any `.tflite`,
  runs N inferences, and prints a machine-parsable report (peak arena,
  per-op timings, p50/p95 latency).
- `firmware/edge-bench/build.sh` ‚Äî clones tflite-micro at the pinned commit,
  runs `make microlite` for `TARGET=linux`, and links `build/edgebench` via
  `Makefile.bench`.
- `firmware/edge-bench/run_bench.py` ‚Äî Python wrapper that invokes the
  binary, parses stdout, applies the documented x86‚ÜíESP32-S3 multiplier,
  and writes `training/edge/results/<story-id>-tflm.json`.
- `training/edge/tests/test_tflm_bench_parser.py` ‚Äî 11 parser unit tests
  (mocks `subprocess.run`, asserts JSON shape and percent rollup).
- `firmware/edge-bench/README.md` ‚Äî TFLM commit pin, multiplier rationale
  with three citations, output schema, TFLM-compat caveats.

## Pinned upstream

`tensorflow/tflite-micro @ 51bee03bed4776f1de88dd87226ff8c260f88e3c`
(latest `main` at the time of US-010). Recorded in both
`firmware/edge-bench/build.sh` (`TFLM_COMMIT`) and `README.md`.

## x86 ‚Üí ESP32-S3 multiplier

Default `8.0`, configurable via `run_bench.py --multiplier`. Citation chain
is in `firmware/edge-bench/README.md` ("Multiplier rationale"); summarised:

1. TFLM `tensorflow/lite/micro/docs/benchmarks.md` ‚Äî INT8 vision workloads
   are 5x‚Äì10x slower on Xtensa LX7 (S3 240 MHz) than on modern x86 desktop.
2. ESP-NN-accelerated kernels close some Conv2D / DepthwiseConv2D gap but
   not all op types.
3. ESP-IDF `cat_face_detect` example reports 6x‚Äì10x slowdowns vs x86.

This is an order-of-magnitude estimate, not a calibrated number. US-011's
markdown report restates the multiplier and its provenance for every row;
once real S3 silicon is in hand, the multiplier should be replaced with the
measured ratio.

## Caveats

- `unzip` is not present on this WSL2 dev box (and on some CI containers);
  `firmware/edge-bench/tools/unzip` is a python shim that the build.sh
  prepends to `PATH` when `/usr/bin/unzip` is missing. The shim handles
  exactly the invocations that TFLM's `download_and_extract.sh` uses.
- TFLM's gen path is `linux_x86_64_default_gcc` on current `main` (the
  `_gcc` toolchain suffix). `build.sh` sniffs for both that and the legacy
  `linux_x86_64_default` path.
- Dependencies fetched by TFLM's Makefile (gemmlowp, ruy, flatbuffers,
  kissfft, pigweed) come from public archives. The Makefile retries on
  transient 502s; in tests, gemmlowp's GitHub archive intermittently
  502'd and recovered. Cached downloads under
  `third_party/tflite-micro/tensorflow/lite/micro/tools/make/downloads/`
  make subsequent builds offline-friendly.
- Latency on the hello_world model is below the timer resolution
  (`raw_x86_us_p50 = 0`); that's expected for a 3-FullyConnected model.
  The real candidates (yolov8n_cat_int8, nanodet_cat_0.5x_224_int8) have
  multi-millisecond latencies and are the intended subjects of US-011.
