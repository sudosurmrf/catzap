# US-007 — NanoDet-Plus pretrained baseline through the eval harness

**Status**: passing.
**Model**: `nanodet-plus-m_416` (1.0x at 416×416), upstream pretrained on COCO.
**Backend**: `onnxruntime` (the main env); the `.pth` route is wired but
requires the isolated venv from
[`training/edge/nanodet/setup_venv.sh`](../nanodet/setup_venv.sh).
**Val set**: `training/edge/data/labeled/val/` (120 frames, the same US-002
held-out split YOLOv8n was evaluated on).

## Headline finding

NanoDet-Plus-m at 416 (1.0x, COCO-pretrained, **never trained on cats**)
scores **mAP@0.5 = 0.704** on our 120-image cat-only val — **beating** stock
YOLOv8n at 224 (0.636) on the same set, despite being a smaller model
(1.19 M vs 3.16 M params, 4.79 MB vs 6.20 MB on disk). The PRD predicted
"substantially lower than YOLOv8n baseline (because NanoDet is general-purpose
COCO and not cat-finetuned)" — that prediction is **not borne out**. NanoDet's
larger input (416 vs 224) and higher detection-density head (3598 priors)
appear to compensate for the small backbone. The fine-tuned YOLOv8n still
dominates (0.909) but the gap to NanoDet pretrained is only 20.5 mAP-points.

This shifts the narrative for US-008 (NanoDet fine-tune): a tighter race than
expected, because NanoDet starts strong. If US-008 lifts NanoDet's cat-only
mAP by ~10 points it will be competitive with — or beat — the YOLOv8n track
once both are quantized.

## Latency caveat

NanoDet runs at 416×416 vs YOLOv8n's 224×224, so the 26.6 ms vs 5.9 ms
single-thread CPU latency is NOT directly comparable. US-008 fine-tunes
NanoDet at 224×224 (the input-spec target) which should bring its latency
into the same neighborhood as YOLOv8n. Per-pixel work scales roughly with
spatial area, so 224² / 416² ≈ 0.29 → expected ~7-8 ms p50 at 224, putting
NanoDet within striking distance of YOLOv8n's 5.9 ms.

## Deviation: 0.5x pretrained checkpoint not published

The PRD acceptance criterion says:

> Download the upstream NanoDet-Plus-m 0.5x pretrained 416x416 checkpoint to
> `training/edge/nanodet/checkpoints/nanodet_plus_m_0.5x_pretrained.pth` (use
> the official release URL; document SHA256).

Upstream's GitHub releases publish only `nanodet-plus-m` (1.0x) and
`nanodet-plus-m-1.5x` at 320 / 416. The 0.5x variant exists in the legacy
v0.x config tree (`config/legacy_v0.x_configs/nanodet-m-0.5x.yml`) but no
matching pretrained `.pth` is released. So:

* US-007 evaluates the closest-published baseline,
  `nanodet-plus-m_416_checkpoint.ckpt` / `nanodet-plus-m_416.onnx` (1.0x at
  416×416).
* The canonical filename
  `training/edge/nanodet/checkpoints/nanodet_plus_m_0.5x_pretrained.onnx` is
  preserved as a symlink so the PRD-shaped command line resolves; the file
  pointed at is the 1.0x baseline.
* US-008 will fine-tune the 0.5x architecture from scratch (or 1.0x-init →
  width-scale to 0.5x) with the auto-labeled cat dataset. The decision of
  whether to keep the 0.5x or fall back to 1.0x is deferred to US-008.

The deviation matches the pattern set by US-002 (live ESP32-CAM stream
unavailable, fell back to data/cat_photos/ bootstrapping) — keep the
research moving forward when an external constraint blocks the spec verbatim.

## Reproduce

ONNX route (main env):

```bash
python -m training.edge.eval.run_eval \
    --model training/edge/nanodet/checkpoints/nanodet_plus_m_0.5x_pretrained.onnx \
    --format pytorch \
    --story-id US-007 \
    --val-dir training/edge/data/labeled/val/ \
    --imgsz 416
```

`run_eval._load_adapter` sniffs the model path and dispatches to
`training/edge/eval/adapters/nanodet_adapter.py` regardless of `--format`,
so the spec command works as written.

PyTorch route (isolated venv):

```bash
training/edge/nanodet/setup_venv.sh
source training/edge/nanodet/.venv/bin/activate
python -m training.edge.eval.run_eval \
    --model training/edge/nanodet/checkpoints/nanodet_plus_m_0.5x_pretrained.pth \
    --format pytorch \
    --story-id US-007 \
    --val-dir training/edge/data/labeled/val/ \
    --imgsz 416
```
