# US-011 — TFLM x86 emulator runs on quantized YOLOv8n-cat and NanoDet-cat

## What this story did

Ran `firmware/edge-bench/run_bench.py` (built in US-010) on every candidate
INT8 `.tflite` produced earlier in the PoC, captured arena / op-breakdown /
host-side latency, and converted x86 latency into a predicted ESP32-S3 latency
via the `8.0x` multiplier documented in `firmware/edge-bench/README.md`.

Per US-011 acceptance, runs include:
- US-004 baseline INT8 (`yolov8n_cat_int8.tflite`)
- US-005 QAT INT8 (`yolov8n_cat_qat_int8.tflite`)
- US-006 distilled INT8 (`yolov8n_cat_distilled_int8.tflite`)
- US-009 NanoDet INT8 (`nanodet_cat_0.5x_224_int8.tflite` — symlink target is
  the pretrained 1.0x INT8 per US-009 deviation)

All commands used `--runs 50` and the `--arena-kb 8192` default (the PSRAM
budget the ESP32-S3 will have). All four loaded cleanly — `tflm_compatible:
true` across the board; the PRD's warning about TFLM-incompatible Detect /
NMS ops did not bite this PoC because both architectures emit post-processing
outside the TFLite graph.

## Headline findings

1. **All four candidates load under TFLM.** No unsupported-op failures, no
   allocate_tensors failures. The arena pre-allocation budget of 8 MB is
   never used — actual `arena_used_bytes` is 677 KB (yolov8n family) and
   2.20 MB (nanodet), both well within the ESP32-S3's ~8 MB PSRAM target.
2. **Predicted S3 fps is below the PRD's 5-fps success target for both
   candidates** (yolov8n ~0.106 fps, nanodet ~0.068 fps under the 8x
   multiplier). This is the load-bearing signal for US-012's recommendation
   section. See "Knock-on for US-012" below.
3. **The three yolov8n variants benchmark identically.** PTQ, QAT, and
   distilled produce byte-identical `.tflite` size (3,220,840 B), identical
   arena (692,816 B), and within-noise host latency (delta ~5 ms p50 across
   variants). This is expected: the export pipeline is the same; only the
   fp32 weights upstream differ. The TFLM bench cannot distinguish them on
   speed — the discriminator is mAP from US-004 / US-005 / US-006.
4. **NanoDet is half the file size but uses ~3.3x the arena and is ~1.55x
   slower at p50.** The arena reflects the larger 416x416 input feature
   maps (per US-007's deviation — no upstream 0.5x pretrained .pth, US-009
   symlinked 1.0x). NanoDet's denser 3598-prior grid plus DEPTHWISE_CONV
   stack accounts for the latency gap.
5. **CONV_2D dominates everywhere (88-93% of inference time).** Optimizing
   on real ESP32-S3 will mean leaning on the ESP-DL / vector instructions
   for INT8 conv; nothing else matters within an order of magnitude. The
   yolov8n family also spends ~6.6% in LOGISTIC (sigmoid output activation)
   which is surprising given how cheap LOGISTIC usually is — likely a
   single-output-tensor artifact applied across all 1029 anchors.

## How this was reproduced

```bash
# US-010 first — builds firmware/edge-bench/build/edgebench
bash firmware/edge-bench/build.sh

# Then for each .tflite, with --runs 50:
python3 firmware/edge-bench/run_bench.py \
    --model training/edge/models/yolov8n_cat_int8.tflite \
    --story-id US-011-yolov8n --runs 50

python3 firmware/edge-bench/run_bench.py \
    --model training/edge/models/yolov8n_cat_qat_int8.tflite \
    --story-id US-011-yolov8n-qat --runs 50

python3 firmware/edge-bench/run_bench.py \
    --model training/edge/models/yolov8n_cat_distilled_int8.tflite \
    --story-id US-011-yolov8n-distilled --runs 50

python3 firmware/edge-bench/run_bench.py \
    --model training/edge/models/nanodet_cat_0.5x_224_int8.tflite \
    --story-id US-011-nanodet --runs 50
```

The four `training/edge/results/US-011-*-tflm.json` files are committed; the
table above is regeneratable from them via `firmware/edge-bench/aggregate.py`.

## TFLM compatibility — checked, no surprises

The PRD warned that YOLO heads and NanoDet's GFL post-processing might trip
TFLM's builtin op resolver. They didn't, because the upstream export
pipelines (`ultralytics .export(format=onnx)` -> onnx2tf -> TFLite, and
`nanodet.tools.export_onnx` -> onnx2tf -> TFLite) emit only the network
forward pass; non-maximum suppression and box decoding sit outside the
`.tflite` graph and are run on the host (Python in our eval harness; will
need to be re-implemented in C on the ESP32-S3). All ops in all four models
are vanilla builtin ops resolvable by TFLM's `MicroMutableOpResolver` —
verified by the absence of any `unsupported` markers in the op_breakdown.

If a future export DOES emit a custom op, `aggregate.py` keeps the row in
the table and surfaces the offending op in the `tflm_compatible` column —
covered by `test_tflm_incompatible_row_is_kept_with_offending_ops_and_tag`.

## Knock-on for US-012

- Both candidate INT8 architectures fit easily within PSRAM (worst case
  nanodet 2.2 MB << 8 MB), so the PRD's "predicted ESP32-S3 tensor arena
  <= 4 MB" success metric is met by both.
- The 8x multiplier is conservative for vector-aware ops on the S3
  (Espressif's ESP-DL claims up to 3-4x speedup over the reference path),
  so US-012 should report both the conservative and the speedup-adjusted
  fps. Even at the optimistic end (3x with ESP-DL = effective multiplier
  ~2.7x), yolov8n hits ~0.31 fps and nanodet ~0.20 fps — still under the
  5-fps target.
- The PRD allows a follow-up if `predicted_fps < 5` AND the use case
  permits it ("acceptable for cat detection given low motion?"). For the
  catzap monitoring rig the cats sit still for minutes; ~0.3 fps is
  defensible if the alternative is "no on-device detection at all". US-012
  will frame this honestly.
- yolov8n is the recommended first-flash candidate based on these numbers
  alone (smaller arena, ~1.55x faster than nanodet); the mAP picture from
  US-006 (KD INT8 = 0.205) is the actual blocker, not the latency.
