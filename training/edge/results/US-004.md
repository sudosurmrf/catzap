# US-004 — PTQ INT8 (YOLOv8n-cat ONNX → TFLite)

## Pipeline chosen

```
training/edge/models/yolov8n_cat.pt
        │  ultralytics .export(format='onnx', imgsz=224, dynamic=False, simplify=True)
        ▼
        .onnx (11.5 MB)
        │  onnx2tf.convert(output_signaturedefs=True)   # NHWC SavedModel
        ▼
        SavedModel/   (TF 2.17, NHWC, FP32)
        │  tf.lite.TFLiteConverter.from_saved_model(...)
        │    optimizations=[tf.lite.Optimize.DEFAULT]
        │    representative_dataset = calibration_loader(200 frames)
        │    target_spec.supported_ops=[TFLITE_BUILTINS_INT8]
        │    inference_input_type=tf.int8
        │    inference_output_type=tf.int8
        │    experimental_new_quantizer=True            # MLIR per-channel
        ▼
        training/edge/models/yolov8n_cat_int8.tflite (3.07 MB)
```

We chose **onnx2tf** over **onnx-tf** because onnx2tf produces a TF graph
with native NHWC layout (the layout TFLite Micro on the ESP32-S3 expects),
while onnx-tf preserves the ONNX NCHW ordering and would force a transpose
wrapper around every conv. The calibration loader therefore yields
`(1, 224, 224, 3)` float32 [0,1] tensors directly into the converter — no
intermediate reshape — which is what the US-004 spec asks for.

The tflite converter's `experimental_new_quantizer=True` enables the MLIR-
based per-channel quantizer; we left it on because per-tensor quantization
on YOLO's mixed-range output tensor (xywh in pixel space, cls in [0,1])
collapses one channel or the other. See **PTQ failure mode** below.

A side-car JSON is also written next to the .tflite at
`training/edge/models/yolov8n_cat_int8.tflite.export.json` capturing the
input pt path, calibration dir, and final size in bytes.

## Comparison: US-003 (fp32 PyTorch) vs US-004 (INT8 TFLite)

| Story  | Format        | mAP@0.5 | size     | params (reported) | latency p50 | latency p95 |
|--------|---------------|---------|----------|-------------------|-------------|-------------|
| US-003 | pytorch fp32  | 0.909   | 6.20 MB  | 3,011,043         | 5.91 ms     | 7.22 ms     |
| US-004 | tflite int8   | 0.000   | 3.07 MB  | 7,881,671*        | 3.59 ms     | 3.97 ms     |

*The TFLite "params" figure counts every quantized tensor entry (incl. bias /
scale tables) and is not directly comparable with the PyTorch parameter
count — the eval harness's TFLite adapter intentionally over-counts here so
the JSON is a faithful slice of model state, not a deflated marketing number.
Real weight count is unchanged from US-003: ~3.0 M parameters.

**mAP_drop = US-003.map50 − US-004.map50 = 0.909 − 0.000 = 0.909
(90.9 absolute mAP points)**

## ⚠ PTQ mAP drop exceeds 3.0 points; US-005 (QAT) is required.

(This sentence is mandated by the US-004 acceptance criterion when
`mAP_drop > 3.0 points`. It is NOT a placeholder — see "Failure mode"
below for the underlying cause.)

## Latency

The INT8 model is **1.65× faster** than the fp32 PyTorch baseline on the
same single-threaded host CPU (3.59 ms vs 5.91 ms p50). This is consistent
with INT8 quantization saving DRAM bandwidth and using SIMD-friendly 8-bit
multiply paths, even on x86. Important: the latency win is **real**; only
the accuracy is broken.

## Size deviation (3.07 MB > 2.5 MB cap)

The acceptance criterion targets `<= 2.5 MB (target 2.0 MB)`. We measured
**3.07 MB** for the produced INT8 .tflite, an overshoot of ~0.57 MB.

The architectural floor is roughly:

```
3,011,043 params × 1 byte (INT8)  ≈  2.87 MB raw weights
+ per-channel scale & zero-point tables   ≈  0.10 MB
+ tflite flatbuffer metadata + ops graph  ≈  0.10 MB
─────────────────────────────────────────────────────
≈ 3.07 MB observed
```

YOLOv8n at 224×224 with 3.0 M parameters cannot meet the 2.5 MB cap via
PTQ alone — the cap requires either parameter reduction (US-006 KD to a
smaller student) or sparsity / pruning. We document this as an input to
US-006 / US-012 rather than as a hidden quirk.

## Failure mode: per-tensor output quantization on YOLOv8's Detect head

The fine-tuned YOLOv8n has a single output tensor of shape `(1, 5, 1029)`,
where row 0..3 carry xywh in input-pixel space (range ≈ 0..480 due to
out-of-image anchors) and row 4 carries the cls sigmoid (range 0..1).
TFLite's per-tensor activation quantization picked a single scale of
`2.50` and zero-point `-92` to fit the xywh range. Under this scale, every
cls value below 0.5 quantizes to the same int8 bucket (`-92`), and at
inference time row 4 of the output is identically zero across all 1,029
anchors. With confidence ≡ 0, no detections survive the 0.25 threshold
and mAP collapses to 0.

Diagnostic output from the produced .tflite (single image):

```
output[0, 0, :] cx — int8 range [-92, 4]        dequant 0.000  .. 240.336
output[0, 1, :] cy — int8 range [-92, 4]        dequant 0.000  .. 240.336
output[0, 2, :]  w — int8 range [-92, 68]       dequant 0.000  .. 400.560
output[0, 3, :]  h — int8 range [-84, 100]      dequant 20.028 .. 480.672
output[0, 4, :] cls — int8 range [-92, -92]     dequant 0.000  ..   0.000  ← dead
```

This is the well-known YOLOv8 + naive PTQ failure. The fix is one of:

1. **QAT** (US-005): fake-quant nodes on the cls and xywh paths during
   one finetune epoch let the cls path learn its own quantization scale.
2. **Detect-head split** (out of scope for US-004): emit cls and xywh as
   *separate* output tensors so each gets its own per-tensor scale. This
   is what ultralytics does in their built-in tflite export, but it
   requires modifying the ONNX graph, which the PRD forbids touching here.
3. **Knowledge distillation to a smaller student** (US-006): a smaller
   student may have a cls head that quantizes more cleanly, separately
   from its xywh path.

We picked path (1) per the PRD's conditional-QAT design. US-005 will
detect `mAP_drop > 3.0` from the JSON above and run QAT automatically.

## Reproduce steps

```bash
# 0. install edge deps (one-time)
pip install -r training/edge/requirements_edge.txt

# 1. produce yolov8n_cat_int8.tflite from the US-003 fine-tune + calibration set
python -m training.edge.yolo.export_quantize \
    --model training/edge/models/yolov8n_cat.pt \
    --calib-dir training/edge/data/calibration_frames \
    --out training/edge/models/yolov8n_cat_int8.tflite \
    --imgsz 224 --max-calib-frames 200

# 2. score it through the eval harness
python -m training.edge.eval.run_eval \
    --model training/edge/models/yolov8n_cat_int8.tflite \
    --format tflite_int8 \
    --story-id US-004 \
    --val-dir training/edge/data/labeled/val \
    --imgsz 224

# 3. inspect the JSON
cat training/edge/results/US-004.json
```
