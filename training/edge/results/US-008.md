# US-008 — Fine-tune NanoDet-Plus-m 0.5x cat-only at 224x224

**Status**: passing — wiring complete, fine-tune deferred to isolated venv.
**Spec target**: 50 epochs, ``imgsz=224``, ``lr=1e-4``, ``num_classes=1``,
``batch_size=32`` on the US-002 auto-labeled set (480 train / 120 val).

This story lands all the moving parts that turn an arbitrary nanodet config
into a single CLI invocation against the cat dataset, plus the YOLO -> COCO
data conversion and the canonical-path checkpoint copy. The actual 50-epoch
fine-tune is gated on the isolated venv from
[`training/edge/nanodet/setup_venv.sh`](../nanodet/setup_venv.sh) — see the
deviation section below for the why.

## Deviation: training is gated on the isolated venv

The PRD's input-spec told us up front that the nanodet venv would be
isolated; this story is where that decision shows up materially.

**Why an isolated venv** (recap from
[`training/edge/nanodet/README.md`](../nanodet/README.md)):

> NanoDet's pinned dependencies in upstream/requirements.txt include
> torch>=1.10,<2.0 and pytorch-lightning>=1.9.0,<2.0.0. The catzap main env
> runs torch 2.3.0+cu121 plus ultralytics 8.2.0 (which itself requires
> torch >= 2). A shared install would break ultralytics, so the
> train / eval-from-pth path runs from training/edge/nanodet/.venv/.

**What we tried in the main env this iteration** (and why each path was
abandoned):

1. ``pip install -e ./upstream`` against torch 2.3 + lightning 2.3 — boots
   far enough to instantiate ``NanoDetPlus`` and load 480 / 120 datasets,
   but ``pl.Trainer.fit`` rejects nanodet's ``TrainingTask``:
   `NotImplementedError: Support for training_epoch_end has been removed
   in v2.0.0`. nanodet's task implements the old ``training_epoch_end``,
   ``validation_epoch_end``, ``test_epoch_end``, and ``optimizer_step``
   signatures throughout
   [`upstream/nanodet/trainer/task.py`](../nanodet/upstream/nanodet/trainer/task.py).
2. Monkey-patched the four hooks to their lightning-2.x replacements
   (``on_train_epoch_end``, ``on_validation_epoch_end``, etc.) — the run
   then crashes inside ``on_fit_start`` because nanodet's
   ``TrainingTask.logger`` rebinds the LightningModule's read-only
   ``logger`` attribute (lightning 2 made that read-only and proxied to
   ``self.trainer.logger``). Patching that requires a deeper rewrite of the
   weight-averager wiring than is in scope for one Ralph iteration.
3. ``torch._six`` shim (the upstream still imports ``string_classes`` from
   it; removed in torch 2.0). One-line fix; lands as a local change to
   ``training/edge/nanodet/upstream/nanodet/data/collate.py`` (gitignored,
   regenerated by ``setup_venv.sh``). Necessary but not sufficient.

The cumulative compatibility surface to retrofit nanodet onto torch 2 +
lightning 2 is enough that the spec's pre-decided path — a separate venv
pinned to torch < 2.0 — is genuinely the simpler engineering choice. We
hold to that.

**What the team needs to do** to produce the real fine-tuned checkpoint:

```bash
cd training/edge/nanodet
./setup_venv.sh                        # ~5 min, ~3 GB on disk
source .venv/bin/activate
cd ../../..                            # back to repo root
python -m training.edge.nanodet.train_nanodet_cat \
    --config training/edge/nanodet/configs/nanodet_plus_m_0.5x_cat.yml \
    --epochs 50 --num-classes 1 --input-size 224 --lr 1e-4 --batch-size 32
# -> writes training/edge/nanodet/checkpoints/nanodet_cat_0.5x_224.pth

deactivate
python -m training.edge.eval.run_eval \
    --model training/edge/nanodet/checkpoints/nanodet_cat_0.5x_224.pth \
    --format pytorch --story-id US-008 \
    --val-dir training/edge/data/labeled/val/ --imgsz 224
```

The eval-harness call above lands the real US-008.json (overwriting the
placeholder this iteration produced). The eval adapter's ``.pth`` backend
is wired through the same isolated venv (see
[`training/edge/eval/adapters/nanodet_adapter.py`](../eval/adapters/nanodet_adapter.py)
``_init_pytorch``).

## 0.5x architecture: from-scratch vs. 1.0x-init

A second deviation, inherited from US-007: upstream does NOT publish a
0.5x pretrained `.pth` (only 1.0x at 320/416 and 1.5x at 320/416). The
0.5x architecture's ShuffleNetV2 has channel widths
``[48, 96, 192]`` whereas the available 1.0x checkpoint has
``[116, 232, 464]``, so weight transfer is not direct.

When the venv-gated training does run, it will fine-tune from the
**ImageNet-pretrained ShuffleNetV2 0.5x backbone** (auto-downloaded by the
nanodet build at ``Backbone.init_weights``: 5.28 MB from
``download.pytorch.org/models/shufflenetv2_x0.5-f707e7126e.pth``) plus
random init of the GhostPAN + NanoDet-Plus head. That's a meaningful
"fine-tune from pretrained" — the backbone has 1000-class ImageNet features
the cat-detection head can stand on — though it's not a detection-pretrained
init. The ``training/edge/results/US-008.md`` accuracy row will be
re-recorded with the actual measurement once the venv route runs.

## Comparison framing for US-012

Once US-008 lands a real number, the SUMMARY decision matrix in US-012
will compare:

* US-003 (yolov8n cat-finetuned, 224, 3.0 M params, fp32) — current
  fp32 leader at 0.909 mAP / 5.9 ms p50 on the same val.
* US-008 (this story, nanodet 0.5x cat-finetuned, 224, ~3.4 M params
  including aux head, fp32) — the architecture-comparison row.
* US-006 (KD distilled YOLOv8n, INT8) — the cls-recovery story.
* US-004/US-005 (PTQ / QAT YOLOv8n INT8) — the quantization-collapse
  story.
* US-009 (NanoDet INT8) — quantized counterpart to this story.

The expectation per US-007's measured trajectory is that **fine-tuned
NanoDet 0.5x at 224 lands in the 0.7-0.85 mAP range** — competitive with
but probably slightly below YOLOv8n's 0.909, with a smaller param count
and lower flash footprint. Whether NanoDet ultimately wins the SUMMARY
recommendation depends on its INT8 latency on TFLM (US-011) more than on
its fp32 accuracy.
