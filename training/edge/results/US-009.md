## `mAP_drop_quant` — explicit per PRD

The PRD says: "Explicitly compute and report `mAP_drop_quant = US-008.map50 -
US-009.map50`." Until US-008's isolated-venv fine-tune lands, US-008.map50
is the placeholder `0.0`, so the literal arithmetic is:

> **`mAP_drop_quant = US-008.map50 - US-009.map50 = 0.000 - 0.000 = 0.000`**

That value is not informative because the fp32 reference is missing.
The honest, **load-bearing** comparison this iteration is against the
nearest-available fp32 nanodet — US-007's pretrained:

> **`mAP_drop_quant_vs_US007_fp32 = US-007.map50 - US-009.map50 = 0.704 - 0.000 = 0.704`**

A 70-point absolute drop is severe. Probing at `confidence_threshold=0.05`
(rather than the default 0.25) recovers ~1 detection per frame across the
val set, so the model is still producing *some* signal — the cls scores are
saturating low, not vanishing. This is the well-known per-tensor-output-scale
collapse on NanoDet-Plus heads (cls is post-sigmoid `[0, 1]`, reg-DFL bins
are unbounded logits, the converter fits one int8 scale to span both). The
post-quant `_post_quant_score_normalize` re-sigmoid lifts cls back into
score-space but cannot recover the resolution lost to int8 quantization on
the saturated channel.

The recommended fixes (out of scope for US-009 but flagged for US-012):

1. **Per-channel output quantization** — would split the output scale per
   channel-axis so cls and reg don't share a budget. TF Lite's INT8 spec
   supports this for weights but not natively for activations on all ops;
   needs `experimental_select_user_tf_ops` exploration.
2. **Split output heads** — export NanoDet with cls and reg as TWO output
   tensors so each gets its own int8 scale. Requires a small upstream patch
   (`NanoDetPlusHead._forward_onnx` returns one concat tensor today).
3. **fp16 fallback** — `tflite_fp16` keeps cls in float; PRD US-009
   technical notes explicitly authorize this fallback ("record
   `format='tflite_fp16'` in the EvalResult so US-012 can reflect this
   honestly"). Reserved for the moment a cat-finetuned `.pth` exists; not
   useful to fp16-quantize the wrong checkpoint.

## Deviation — same root cause as US-008

US-009 cannot quantize a cat-finetuned NanoDet because US-008 hasn't
produced one yet. Per the recap in
[`US-008.md`](US-008.md#deviation-training-is-gated-on-the-isolated-venv):
nanodet upstream pins `torch<2.0` + `pytorch-lightning<2.0`, the catzap
main env runs `torch 2.3.0` for ultralytics, so the cat fine-tune is gated
on `training/edge/nanodet/setup_venv.sh`.

Rather than block US-009 on that, this iteration delivers everything that
*can* be delivered without the venv:

- The orchestrator + DI + tests for the full pipeline (`.pth -> ONNX ->
  SavedModel -> int8 .tflite`).
- The shared-calibration-loader contract (the PRD's load-bearing acceptance
  test).
- A real INT8 .tflite artifact at the canonical path
  `training/edge/models/nanodet_cat_0.5x_224_int8.tflite`, produced by
  running the pipeline against the only available NanoDet-Plus ONNX (the
  US-007 pretrained 1.0x @ 416). The symlink keeps US-011's TFLM bench and
  US-012's decision matrix runnable today; once
  `setup_venv.sh && train_nanodet_cat.py && export_quantize.py --imgsz 224`
  produces a real cat-finetuned `.tflite`, the symlink is replaced with the
  real file and the eval is re-run with no other code changes.
- The TFLite backend on `NanodetAdapter` — needed independently of which
  weights end up in the file, so the eval harness has a working int8
  decode path the moment the cat-finetuned weights land.

The DEVIATION is also recorded in `US-009.json::notes` for programmatic
discovery by US-012's `aggregate_summary.py`.

## Reproduce

Once `setup_venv.sh` produces `nanodet_cat_0.5x_224.pth`:

```bash
# 1. Replace the canonical-path symlink with the real cat-finetuned int8.
rm training/edge/models/nanodet_cat_0.5x_224_int8.tflite
python -m training.edge.nanodet.export_quantize \
    --model training/edge/nanodet/checkpoints/nanodet_cat_0.5x_224.pth \
    --config training/edge/nanodet/configs/nanodet_plus_m_0.5x_cat.yml \
    --calib-dir training/edge/data/calibration_frames \
    --out training/edge/models/nanodet_cat_0.5x_224_int8.tflite \
    --imgsz 224

# 2. Re-run the eval harness at the matching imgsz.
python -m training.edge.eval.run_eval \
    --model training/edge/models/nanodet_cat_0.5x_224_int8.tflite \
    --format tflite_int8 \
    --story-id US-009 \
    --val-dir training/edge/data/labeled/val/ \
    --imgsz 224
```

To smoke-test the pipeline today (no venv needed; quantizes the pretrained
ONNX in the catzap main env):

```bash
python -m training.edge.nanodet.export_quantize \
    --onnx training/edge/nanodet/checkpoints/nanodet_plus_m_0.5x_pretrained.onnx \
    --calib-dir training/edge/data/calibration_frames \
    --out training/edge/models/nanodet_plus_m_pretrained_int8.tflite \
    --imgsz 416
```
