## Codebase Patterns

### Off-graph NMS verifier scans op_breakdown, not the .tflite schema
- Where: `training/edge/eval/decode_only.py:verify_off_graph` +
  `training/edge/eval/decode_only.py:IN_GRAPH_NMS_OPS` +
  `training/edge/tests/test_decode_only.py` (3 verifier tests)
- Pattern: rather than parsing the TFLite flatbuffer schema to look for
  post-processing ops, scan the op_breakdown the existing edge-bench
  TimingProfiler already emits. The four canonical "in-graph NMS" op
  names (TFLite_Detection_PostProcess for SSD, NON_MAX_SUPPRESSION_V4/V5,
  and TF's CombinedNonMaxSuppression) cover every exporter we've seen.
  This re-uses the runs=N TFLM bench as the source of truth — no
  duplicate flatbuffer parsing, and the verifier produces the same
  result regardless of whether the model came from ultralytics ONNX
  export, onnx2tf, or a hypothetical future tf.lite-direct path. If a
  future iteration adds a NEW post-processing op name, add it to
  `IN_GRAPH_NMS_OPS` (the regression test
  `test_in_graph_nms_ops_set_covers_known_exporter_outputs` will catch
  removals).

### onnx2tf YOLOv8 .tflites strip post-processing entirely
- Where: `training/edge/results/iter-G.md` ("Verifier output" — 16 ops,
  zero NMS) + `training/edge/results/iter-G_off_graph_nms.json:verifier`
- Pattern: every YOLOv8 .tflite produced via the v2 PTQ pipeline
  (ultralytics ONNX export → onnx2tf SavedModel → TFLiteConverter)
  emits backbone + neck + Detect-head convs ending in a single fp32
  DEQUANTIZE on the `(1, 5, 1029)` cls/xywh tensor. Decode (xywh /
  imgsz, threshold, xyxy, clip) and NMS are NOT in the graph — they
  must run host-side in firmware C. iter-G's runs=200 op_breakdown
  records this for iter-A's tflite (CONV_2D / LOGISTIC / ADD / MUL /
  PAD / MAX_POOL_2D / RESIZE_NEAREST_NEIGHBOR / RESHAPE / TRANSPOSE /
  SPLIT / CONCATENATION / STRIDED_SLICE / SUB / SOFTMAX / QUANTIZE /
  DEQUANTIZE — 16 ops, none of them NMS). Same shape holds for iter-D
  / iter-E candidates that share the export pipeline. Future stories
  swapping to a different export (ultralytics' built-in `format='tflite'`
  with `nms=True`, or any TF SavedModel containing
  `tf.image.combined_non_max_suppression`) MUST re-verify.

### Decode + NMS post-process budget is negligible vs YOLOv8n inference
- Where: `training/edge/results/iter-G.md` ("Post-process budget") +
  `training/edge/results/iter-G_off_graph_nms.json:post_process_budget`
- Pattern: at imgsz=224 YOLOv8n emits 1,029 anchors. Decode is one
  linear scan with ~6 FLOPs/anchor (max(cls), threshold, xywh/imgsz,
  xywh→xyxy, clip) and NMS is O(K^2) on whatever survives the
  threshold (typically <5 boxes for single-class cat detection).
  Estimated cost on ESP32-P4 without ESP-DL: ~0.0154 ms total, vs
  ~5,785 ms inference (~0.0003%). iter-H's SUMMARY_v2 can cite
  `predicted_p4_ms_p50` directly as the v2 frontier latency without
  budgeting extra time for off-graph post-process.

### runs=200 edge-bench drift vs runs=50 is sub-1%
- Where: `training/edge/results/iter-G_off_graph_nms.json:drift_vs_runs50`
- Pattern: re-running iter-A's TFLM bench at 200 samples (vs 50 in
  iter-A through iter-F) shifts p50 by -0.62% and p95 by -0.01% on
  this hardware. Tight enough that the Pareto verdicts in iter-A
  through iter-F do NOT need re-evaluation against runs=200 samples;
  iter-H can cite the runs=50 numbers uniformly. Future iterations
  that need especially tight p50/p95 (e.g. comparing two latency-
  matched candidates at the 1% margin) should bump runs to 200, but
  it's not a blanket requirement.

### iter-A's float-output dequant works on YOLOv8 but NOT NanoDet — predict before applying
- Where: `training/edge/results/iter-F.md` ("Why YOLOv8 recovered and NanoDet did not")
  + `training/edge/nanodet/per_channel_quant.py` (thin wrapper that proves the recipe is reusable)
  + `training/edge/results/iter-F_nanodet_per_channel.json:pareto_vs_nanodet`
- Pattern: the `inference_output_type=tf.float32` fix that recovered
  YOLOv8 cls collapse (iter-A: mAP 0->0.215) recovers cls precision in
  the head's LAST-int8 tensor by deferring dequant to fp32. It does
  NOT recover precision lost UPSTREAM in the head's INT8 conv layers.
  YOLOv8 KD-distilled weights carry calibrated cls magnitudes such that
  the last-int8 tensor preserves enough cls signal; NanoDet-Plus ships
  pretrained on COCO (no cat KD) and emits cls already-sigmoided
  alongside unbounded reg-DFL logits, so a per-tensor activation scale
  on the head's conv outputs is dominated by reg-DFL and quantizes cls
  AND reg-DFL into near-zero buckets BEFORE the dequant. Diagnostic:
  every prior anchor predicts the same constant bbox post-quant. When a
  new architecture lands, predict whether iter-A's recipe will recover
  PTQ collapse by asking: "does the head's last-int8 tensor carry enough
  cls signal to be worth dequantizing?". If no, you need head-split ONNX
  surgery (split cls + reg into separate output tensors so each gets
  its own activation scale) or fp16 fallback. Don't burn a full iter-*
  slot on float-output dequant alone for such architectures.

### Dual-baseline Pareto aggregator pattern (same-arch + cross-arch)
- Where: `training/edge/results/aggregate_iter_f.py:aggregate` (introduces
  `pareto_vs_nanodet` + `pareto_vs_yolo_v2_frontier` keys)
  + `_verdict_against` helper for scoring against an arbitrary baseline JSON
- Pattern: `aggregate_iter_a.py` had ONE baseline; iter-F introduced TWO
  because the candidate is being scored against (a) its own architectural
  predecessor (US-009 NanoDet) AND (b) the cross-architecture frontier
  (iter-A YOLOv8). Both verdicts in one JSON: `pareto_vs_<X>` keys, same
  schema as the existing `pareto` key but explicitly labeled. This lets
  iter-H's SUMMARY_v2 ingest cross-architecture comparisons without re-
  running aggregators. Any future iter-* candidate that's worth comparing
  against multiple frontiers should follow this shape.

### iter-E finding: narrowing student width WITHOUT INT16 activations re-triggers cls collapse
- Where: `training/edge/results/iter-E.md` ("Why mAP collapsed at PTQ")
  + `training/edge/results/iter-E_yolov8n_0p75x.json:pareto.deltas`
- Pattern: KD distillation at 0.75x effective width (= 0.1875 stamped
  vs YOLOv8 base; ~57% of yolov8n params) reproduces iter-A's PTQ
  pipeline byte-for-byte but mAP collapses 0.215 -> 0.000 because the
  narrower cls activations carry less magnitude than the per-tensor
  quantizer can preserve. Three independent confirmations now (v1
  US-004, iter-C finetune-only, iter-E narrower student) — KD-shaped
  cls signal at YOLOv8n full width is the MINIMUM viable bar for
  iter-A's PTQ pipeline to work. Any future "make YOLOv8 smaller" lever
  must either stack iter-D's INT16x8 quant on top, surgically split the
  Detect head, or accept mAP=0 under the per-channel-friendly recipe.

### width_multiple is a 2-space variable: user-facing vs yaml-stamped
- Where: `training/edge/yolo/train_yolov8n_0p75x_cat.py:regenerate_config`
  + `test_regenerate_config_composes_with_yolov8n_base`
- Pattern: ultralytics' `parse_model` interprets the yaml's
  `width_multiple` as a fraction of YOLOv8 BASE channel widths, NOT
  yolov8n's. yolov8n's effective width is already 0.25 (vs YOLOv8 base);
  yolov8s is 0.5; yolov8m is 0.75. So a "0.75x of yolov8n" model
  actually wants `width_multiple = 0.25 * 0.75 = 0.1875` stamped into
  the yaml. The iter-E wrapper exposes the user-facing 0.75 on the CLI
  and composes it with yolov8n's 0.25 base before stamping. Surface the
  user-facing knob in any rollup JSON's audit trail (the yaml number is
  not what the user typed). Same pattern applies for `depth_multiple`
  (yolov8n base = 0.33, iter-E default user-facing = 1.0 -> stamped
  0.33 = unchanged).

### KD without .pt round-trip: patch_student_loss_for_distill on live instance
- Where: `training/edge/yolo/train_yolov8n_0p75x_cat.py:_default_distill_train_fn`
  + `training/edge/yolo/distill_train.py:patch_student_loss_for_distill`
- Pattern: `train_distilled` (US-006) takes a `student_pt: Path` because
  it reloads via `YOLO(student_pt)` and trains. YAML-built models have
  no populated `self.ckpt` dict, so `student.save()` cannot round-trip
  the partial-loaded weights — calling `train_distilled(student_pt=...)`
  on a saved-yaml-built student loses the partial-load. The fix:
  manually do what `train_distilled` does internally — instantiate a
  teacher YOLO, register `patch_student_loss_for_distill` on the LIVE
  student via `student.add_callback("on_train_start", _on_train_start)`,
  then call `student.train(...)` directly. Net effect is byte-identical
  to `train_distilled` minus the .pt round-trip. Reusable for any
  future YAML-built / partially-initialized / distilled training flow.

### partial_load_yolov8n_weights: slice donor channels to fit narrower student
- Where: `training/edge/yolo/train_yolov8n_0p75x_cat.py:partial_load_yolov8n_weights`
- Pattern: when initializing a narrower student from a wider donor
  (yolov8n -> 0.75x; yolov8s -> yolov8n; etc.), strict
  `load_state_dict` fails on every Conv2d. Walk donor.state_dict() and
  for each matched key, slice the donor tensor to the student's smaller
  shape along ALL leading axes (`tuple(slice(0, s) for s in
  student_tensor.shape)`). Returns a `summary` dict with `loaded_keys
  / matched_keys_total / skipped_missing_in_student /
  skipped_mismatched_shape / fallback_random_init / error?` so iter-H
  / future code can audit how much of the init is real vs random. On
  the iter-E run: 355/355 keys all loaded successfully — slicing
  works for every conv when the donor is wider on every axis. If donor
  load fails at the YOLO() step (file missing / corrupt), gracefully
  falls back to random init with `fallback_random_init=True` and an
  `error` field — per the AC: "from-scratch if the partial load fails".

### iter-D finding: INT16 activations stack with iter-A float-output for huge mAP recovery
- Where: `training/edge/yolo/mixed_int_quant.py` +
  `training/edge/results/iter-D.md` ("Why this works")
- Pattern: iter-A's float32 output dequant fixed cls collapse on the
  output side (one DEQUANTIZE op after the head). iter-D is the
  orthogonal internal-side fix: promote ALL activations to INT16 via
  `EXPERIMENTAL_TFLITE_BUILTINS_ACTIVATIONS_INT16_WEIGHTS_INT8`. INT16
  has 256x the dynamic range of INT8, so the head's pre-output
  `(1, 5, 1029)` activation tensor can carry xywh and cls under one
  per-tensor scale without crushing cls. Stacked, iter-A + iter-D
  recover +0.273 absolute mAP (0.215 -> 0.488) over iter-A alone.
  Cost: arena 2x (INT16 activations are 2x int8); P4 latency
  unchanged on TFLM reference path. Weights stay INT8 / per-channel
  so on-disk size is +2.4% only.

### Mixed INT8/INT16 model is TFLM-compatible (op coverage held)
- Where: `training/edge/results/iter-D-tflm.json:tflm_compatible` +
  `training/edge/results/iter-D.md` ("TFLM compatibility")
- Pattern: PRD's technical notes warned that the INT16x8 path might
  trip TFLM's incomplete INT16 op coverage. Real result: every
  INT16-activation op the converter emits (CONV_2D, MUL, LOGISTIC,
  ADD, PAD, SOFTMAX, SPLIT, ...) IS implemented in TFLM's standard
  op resolver. `allocate_tensors_status=ok`,
  `tflm_compatible=true`. CONV_2D shifts from 92.5% (iter-A) to
  98.83% (iter-D) of total inference time — INT16 raises CONV_2D's
  compute cost more than proportionally vs lighter ops, but the
  shift is internal and doesn't break compat. Future iterations that
  need to recover cls precision under INT8-only mandates (e.g.
  ESP-DL targets that don't support INT16 conv) will need the
  head-split path (path (2) in PRD); for the current ESP32-P4
  target the INT16x8 builtin is the simpler path.

### tflite_adapter input cast keys off model dtype, not the is_int8 flag
- Where: `training/edge/eval/adapters/tflite_adapter.py:_preprocess`
- Pattern: the input-dtype branch used to be
  `if in_dt == np.int8 OR self.is_int8`. iter-D's float-IO INT16x8
  model exposes float32 input even though it's quantized internally,
  and the OR condition forced an int8 cast that crashed the
  interpreter. Fix: respect `input_details[0]['dtype']` (the
  authoritative source from `allocate_tensors()`) only. The
  `is_int8` flag is now purely informational — it does NOT force a
  cast. Behavior preserved exactly for every v1 / iter-A INT8 model
  (their `in_dt` is already int8, so the int8 branch still fires);
  iter-A's mAP=0.215478 reproduces bit-exact post-tweak. Future
  formats with float-or-int16 IO get adapter-compat for free.

### iter-C finding: soft prune + supervised-only finetune erodes KD cls signal
- Where: `training/edge/results/iter-C.md` ("Why mAP collapsed at PTQ —
  the diagnosis") + `training/edge/yolo/prune_channels.py:_default_trainer_fn`
- Pattern: iter-A's mAP=0.215 INT8 only survives because US-006 KD
  shaped the cls logits to a magnitude the per-tensor activation
  quantizer can preserve. Any post-hoc finetune of the iter-A `.pt`
  that uses ONLY supervised loss (no distillation signal) drifts the
  cls activation distribution back toward magnitudes the quantizer
  crushes — re-triggering the v1 PTQ cls collapse pattern. iter-C's
  result (fp32 mAP 0.706, INT8 mAP 0.000) is the empirical proof.
  Future "modify the .pt then re-quantize via iter-A" work (iter-E
  retrain at width=0.75x is OK because it re-runs distill_train; but
  any prune/finetune-only flow should keep KD in the loop).

### iter-C finding: soft prune does NOT shrink TFLite arena or size
- Where: `training/edge/results/iter-C.md` ("Numbers" table)
- Pattern: zeroing 25 % of `Conv2d` output channels via
  `torch.nn.utils.prune.LnStructured` + `prune.remove` produces
  identical .tflite size (within 168 B), identical arena_used_bytes
  (within 128 B), and identical predicted P4 latency (within 0.6 %).
  Reason: the per-channel weight scale tables, the int8 zero points,
  and the convolution Op metadata don't shrink when channels go to
  zero — only the raw weight bytes (which compress, but the
  flatbuffer alignment overhead dominates at this model scale).
  Realizing soft-prune as a real efficiency lever requires hard
  channel removal + Detect-head re-init (out of scope for iter-C;
  candidate for a follow-on dedicated story).

### ultralytics `device='auto'` literal fails on CUDA-less hosts
- Where: `training/edge/yolo/prune_channels.py:DEFAULT_DEVICE` +
  `training/edge/yolo/prune_channels.py:_default_trainer_fn`
- Pattern: ultralytics 8.2's `select_device('auto')` raises
  `ValueError: Invalid CUDA 'device=auto' requested` when
  `torch.cuda.is_available()` is False. The two safe defaults are
  `device=''` (empty string lets ultralytics pick CUDA/MPS/CPU
  automatically) or `device='cpu'` (force CPU). iter-C codified
  `DEFAULT_DEVICE = ""` and added a CLI `--device` flag. Existing
  US-003 / US-006 train scripts use `device='auto'` and have so far
  worked because they ran on CUDA-equipped hosts; future stories
  that need to run on a CUDA-less dev box should follow iter-C's
  empty-string convention.

### Sweep orchestrator with per-row failure isolation
- Where: `training/edge/yolo/imgsz_sweep.py:sweep` +
  `training/edge/tests/test_imgsz_sweep.py::test_sweep_continues_past_a_single_quantize_failure`
- Pattern: iter-B/C/E-style "run the same pipeline N times with different
  knobs" orchestrators MUST isolate failures per row. Wrap each of
  quantize / eval / bench in its own `try/except Exception`; on failure,
  set `blocked_reason` and `eval_dict=None`/`tflm_dict=None` for THAT row
  only and continue. Then build the rollup with `build_pareto_verdict`'s
  `blocked_reason=...` short-circuit so the row appears in the JSON with
  `status="blocked"`. Never let a single bad knob abort the rest of the
  sweep. iter-H aggregator counts "every iter-* row that wrote a JSON",
  so silent-drop is much worse than blocked-row.

### Aggregator load_rollups must skip eval/bench sidecars
- Where: `training/edge/results/aggregate_iter_b.py:load_rollups` +
  test_aggregate_iter_b_picks_smallest_qualifying_imgsz
- Pattern: the sweep writes BOTH the rollup `iter-B_imgsz_<N>.json` AND
  per-step sidecars `iter-B_imgsz_<N>_eval.json` /
  `iter-B_imgsz_<N>_bench-tflm.json` (so the inner eval+bench are
  inspectable independently of the rollup). Glob-based loaders must
  filter on a rollup-only key (here: `"imgsz" in d and "candidate_metrics"
  in d`) — without that filter the aggregator KeyErrors on the first
  sidecar. Same gotcha will hit iter-C/E aggregators.

### baseline_metrics uses iter-A's candidate_metrics, NOT US-006-int8
- Where: `training/edge/yolo/imgsz_sweep.py:_baseline_metrics_from_iter_a`
  + `training/edge/results/iter-A_per_channel_quant.json`
- Pattern: iter-B onwards compares against iter-A's `candidate_metrics`
  (its own measured numbers), not against US-006-int8 (iter-A's own
  baseline). This means iter-A is the strict accuracy floor for
  iter-B/C/E even though its Pareto verdict against US-006-int8 was
  `equal`. The aggregate_iter_a.py ingestion path stamps iter-A's
  numbers into `candidate_metrics`, and iter-B reads them via
  `_baseline_metrics_from_iter_a(load_iter_a_aggregate())`. Don't try
  to chase the chain back to US-006 — let iter-A's already-computed
  numbers stand.

### YOLOv8 eval-time imgsz must match training imgsz
- Where: `training/edge/results/iter-B_imgsz_192.json` (mAP=0)
  + `training/edge/results/iter-B_imgsz_320.json` (mAP=-0.016 abs)
- Pattern: re-quantizing a YOLOv8n.pt at eval-time imgsz != train-time
  imgsz (224) collapses cls accuracy. At imgsz=192, mAP drops to 0
  entirely (the model never saw 192-pixel crops, so the head's
  receptive-field math falls apart). At imgsz=256/288/320, mAP holds
  within ~1.7 mAP-points but the arena ALMOST doubles by 320 (532 KB ->
  1.32 MB) and predicted P4 latency more than doubles (4.2s -> 12.1s).
  Conclusion: imgsz reduction is NOT a post-hoc lever — it requires a
  retrain at the target imgsz. iter-E's width=0.75x retrain story is
  the right place to also retrain at smaller imgsz; iter-B's negative
  finding is what justifies prioritizing that.

### iter-* JSON shape: {eval, tflm, pareto, candidate_metrics, baseline_metrics}
- Where: `training/edge/results/aggregate_iter_a.py` +
  `training/edge/yolo/per_channel_quant.build_pareto_verdict`
- Pattern: every v2 follow-on iteration (iter-A..iter-G) emits ONE
  rollup JSON at `training/edge/results/iter-<X>_<slug>.json` whose
  schema is: `{story_id, title, status:"passed"|"blocked",
  blocked_reason, model_path, p4_multiplier, p4_multiplier_source,
  eval:<EvalResult>, tflm:<run_bench shape>, pareto:<verdict-dict>,
  candidate_metrics:{map50,size_bytes,arena_used_bytes,
  predicted_p4_latency_ms_p50,predicted_p4_fps},
  baseline_metrics:{...}}`. The iter-H synthesis script
  (aggregate_summary_v2) ingests these uniformly. Always include all
  fields even on `status="blocked"` so the iter-H matrix can render the
  attempt — never silently drop a blocked iteration.

### build_pareto_verdict: hard-coded thresholds, named-baseline arg
- Where: `training/edge/yolo/per_channel_quant.build_pareto_verdict` +
  module-top constants (`MAP_REGRESS_TOLERANCE`, `LATENCY_IMPROVE_PCT=15.0`,
  `SIZE_IMPROVE_PCT=20.0`, `ARENA_IMPROVE_PCT=15.0`)
- Pattern: Pareto verdict thresholds come from the user message's v2
  follow-on rules — they are NOT runtime knobs. Tests assert the exact
  threshold values so a future iteration can't silently relax them.
  Verdicts: `dominates` (mAP held + 1+ axis past threshold), `equal`
  (mAP held + no axis past threshold), `regress` (mAP dropped),
  `blocked` (caller passes `blocked_reason`). Caller MUST pass a named
  baseline_story (e.g. "US-006-int8") so the verdict is auditable.

### ESP32-P4 latency multiplier: 5.0 (vs S3's 8.0)
- Where: `training/edge/yolo/per_channel_quant.DEFAULT_P4_MULTIPLIER` +
  `_P4_MULTIPLIER_SOURCE` (string)
- Pattern: P4 has 400 MHz dual-core RV32 with PIE/vector ext (vs S3's
  240 MHz single-core for ML). Without ESP-DL acceleration the scalar
  reference path is ~5-6x slower than x86; with vector ext ~3x. We use
  5.0 as the conservative midpoint. Every iter-A..iter-G aggregator
  records its multiplier choice in the JSON's `p4_multiplier` +
  `p4_multiplier_source` fields so iter-H can re-do the math with a
  different multiplier once real boards arrive.

### YOLOv8 PTQ cls-collapse fix #2: float32 OUTPUT dequant
- Where: `training/edge/yolo/per_channel_quant.quantize_to_int8_float_output_tflite`
  + iter-A.md "Why this works"
- Pattern: the cls-collapse documented in v1 progress.txt
  ("YOLOv8 PTQ + per-tensor output quant collapses cls head") has TWO
  orthogonal fixes — (1) US-006 KD which shapes student-side
  activations the calibrator sees, and (2) iter-A's
  `inference_output_type=tf.float32` which appends a DEQUANTIZE op so
  the head reads out at full float resolution. Combined they recover
  0.0 mAP -> 0.215 mAP under PTQ. The architecturally pure fix
  (split Detect head into separate cls/xywh tensors via ONNX surgery)
  is still on the table for a later iteration; for v2 the float-output
  cost (4x larger output_bytes: 5,145 -> 20,580; ~5 µs per dequant on
  TFLM x86) is negligible vs the ~1.16 s total invoke time.

### Bootstrap UUIDs are NOT seeded — val/manifest.txt regen pattern
- Where: `training/edge/auto_label.py:bootstrap` (uuid4().hex calls) +
  `training/edge/data/regenerate_val_manifest.py` (post-bootstrap repair)
- Pattern: v1's progress.txt called the bootstrap "deterministic seed=42"
  but seed=42 only covers IMAGE CONTENT (augmentations / brightness /
  flip). Filenames go through uuid4() which is unseeded — every fresh
  worktree's bootstrap produces a DIFFERENT 600-frame UUID set. v1's
  tracked .txt labels are orphaned because their .jpg counterparts are
  gitignored and unrecoverable. Workaround: run
  `python training/edge/data/regenerate_val_manifest.py` AFTER bootstrap
  + make_dataset_manifest. This rewrites val/manifest.txt to point at
  the freshly-paired files. iter-A through iter-G ALL need this step
  in their reproduce sections. A future iteration could swap uuid4()
  for a seed-derived deterministic ID; out of scope here because doing
  so would orphan v1's already-tracked labels too.

### Decision-matrix aggregator: ROW_SPECS + dual-source ingestion
- Where: `training/edge/results/aggregate_summary.py:ROW_SPECS` +
  `_build_row` + `format_decision_matrix` + `format_recommendation`
- Pattern: when aggregating per-story results into one report, define
  matrix rows as a static `RowSpec(label, eval_json, tflm_json,
  optional)` tuple instead of glob-walking — order is part of the spec
  and required vs. optional rows must be distinguishable. Each row is
  built from up to two source JSONs (EvalResult shape from
  `training/edge/eval/types.py` for accuracy/size/host-CPU latency,
  TFLM shape from `firmware/edge-bench/run_bench.py:report_to_result`
  for arena/predicted_s3_fps/tflm_compatible). The TFLM side is pulled
  through `firmware/edge-bench/aggregate.py:aggregate_row` rather than
  re-parsed locally — single source of truth for the bench-JSON shape.
  Optional rows (e.g. QAT only present if PTQ mAP_drop > 3) are
  silently skipped; required rows missing land in `result.missing` so
  the report can flag them instead of silently rendering an
  incomplete matrix.

### US-011 aggregator is the US-012 ingestion seam
- Where: `firmware/edge-bench/aggregate.py:aggregate_us011_jsons` +
  `:format_comparison_table` + `training/edge/tests/test_us011_aggregate.py`
- Pattern: don't re-parse `run_bench.report_to_result`'s on-disk JSON
  shape from US-012; call `aggregate.aggregate_us011_jsons([...])` to get
  rows (`model`, `size_bytes`, `arena_used_bytes`, `top3_ops`,
  `raw_x86_us_p50`, `predicted_s3_us_p50`, `predicted_s3_fps`,
  `tflm_compatible`, `offending_ops`) and `format_comparison_table(rows)`
  for the markdown rendering. Row order preserved from input. Model name

  `US-011-yolov8n-qat-tflm.json` -> `yolov8n-qat`). Sets the contract
  US-012's `aggregate_summary.py` fills in.

### TFLM x86 build: gen path is `linux_x86_64_default_gcc` on current main

### TFLM make uses /usr/bin/unzip; not present on WSL2 dev box
- Where: `firmware/edge-bench/tools/unzip` + the early-PATH-prepend in
  `firmware/edge-bench/build.sh`
- Pattern: TFLM's `tensorflow/lite/micro/tools/make/download_and_extract.sh`
  shells out to `unzip` to extract `.zip` deps. Many WSL2 / minimal

### TFLM dep downloads (gemmlowp, ruy, kissfft, flatbuffers) flake on 502
- Where: documented in `firmware/edge-bench/README.md` ("TFLM compatibility
  caveats" + the build sequence in `training/edge/results/US-010.md`)
- Pattern: `download_and_extract.sh` retries via curl, but bursts of GitHub

### TFLM MicroProfilerInterface for per-op aggregation
- Where: `firmware/edge-bench/main.cc:TimingProfiler`
- Pattern: `tflite::MicroProfilerInterface` exposes `BeginEvent(tag) ->
  uint32_t` and `EndEvent(handle)` as the public extension point for
  custom timing. Subclassing it (instead of using the bundled
  `MicroProfiler`) is the right path when you need per-tag aggregation
  across multiple Invoke() calls — the bundled class's events_ array is
  private. Use `std::chrono::steady_clock::now()` for ns-precision timing.
  Drop warmup events from aggregation by recording `events_.size()`
  before the timed loop and starting the aggregation walk at that index.

### edgebench stdout: machine-parsable framed key/value blocks
- Where: `firmware/edge-bench/main.cc:main` (printf calls) +
  `firmware/edge-bench/run_bench.py:parse_edgebench_stdout`
- Pattern: print `=== EDGEBENCH START ===` / `=== EDGEBENCH END ===`
  markers around all parser-visible output. Inside, use
  `key: value` lines for scalars and a nested
  `=== OP_BREAKDOWN START ===` / `=== OP_BREAKDOWN END ===` block for the
  per-op rows. Anything outside the START/END markers (e.g. MicroPrintf
  banners) is ignored. New fields are append-only — the parser ignores
  unknown keys. This lets the C++ binary evolve independently of the
  Python schema.

### Per-tensor INT8 output scale collapses NanoDet-Plus cls scores
- Where: `training/edge/eval/adapters/nanodet_adapter.py:_post_quant_score_normalize`
  + `training/edge/results/US-009.md` ("`mAP_drop_quant` — explicit per PRD")
- Pattern: NanoDet-Plus' single output tensor concatenates cls (sigmoid
  `[0, 1]`) with reg-DFL bins (unbounded logits). One per-tensor int8 scale
  has to span both, and reg dominates — cls saturates near zero
  post-dequant. mAP drops from 0.704 fp32 -> 0.000 int8 at default conf.
  Diagnostic: probing at `confidence_threshold=0.05` recovers ~1 det/frame,
  so signal exists, just compressed below the threshold.
  `_post_quant_score_normalize` re-sigmoids when cls dequants out of
  `[0,1]`, but cannot recover resolution lost on a saturated channel.
  US-011 (TFLM bench) and US-012 (decision matrix) MUST flag this for
  NanoDet INT8 — fixes are: per-channel output quant, split output heads
  via upstream patch on `NanoDetPlusHead._forward_onnx`, or fp16 fallback
  (PRD-authorized in US-009 technical notes). YOLO heads don't have this
  pathology because their cls + reg are pre-sigmoid logits at similar scale.

### NanoDet upstream is incompatible with torch>=2 / lightning>=2

### NanoDet-Plus ONNX output decode (self-contained, no nanodet pip)
- Where: `training/edge/eval/adapters/nanodet_adapter.py:decode_nanodet_output`
- Pattern: NanoDet-Plus' ONNX export
  (`nanodet.model.head.NanoDetPlusHead._forward_onnx`) emits a single
  tensor `(1, num_priors, num_classes + 4*(reg_max+1))` with cls already
  sigmoid'd. Decode: split channels, reshape reg to `(N, 4, reg_max+1)`,
  softmax over the bin axis and project onto `[0..reg_max]` (the DFL
  projection), multiply by per-row stride from `_generate_center_priors`,
  apply `distance2bbox(center, [l,t,r,b])`, NMS within the cat (id 15)
  channel, normalize by imgsz. This lets the main env eval pure ONNX with
  NO nanodet pip dependency — the upstream Python package is only needed
  for the .pth backend (lazy-imported, isolated venv).

### NanoDet 0.5x pretrained checkpoint deviation
- Where: `training/edge/nanodet/README.md` ("Pretrained checkpoint" section)
  + `training/edge/results/US-007.md`
- Pattern: upstream's GitHub releases publish only `nanodet-plus-m`
  (1.0x) and `nanodet-plus-m-1.5x` at 320/416 — no 0.5x pretrained .pth
  exists despite the legacy 0.5x configs. US-007's spec command line is
  honored via a symlink at the canonical filename
  `checkpoints/nanodet_plus_m_0.5x_pretrained.{onnx,pth}` pointing at
  `nanodet-plus-m_416_checkpoint.{ckpt,onnx}` (1.0x at 416). US-008 will
  decide whether to truly fine-tune at 0.5x width or stay at 1.0x.

### NanoDet-Plus preprocess uses BGR mean/std on RGB pixels
- Where: `training/edge/eval/adapters/nanodet_adapter.py:_preprocess`
- Pattern: NanoDet's YAML configs list `mean=[103.53, 116.28, 123.675]` and
  `std=[57.375, 57.12, 58.395]` in **BGR** order (matching cv2's default
  channel order), but apply them after a BGR->RGB conversion. Net effect:
  mean/std are reversed before normalization so the values line up with
  the RGB pixel order. Also: pixels stay in `[0, 255]` (NO divide-by-255
  step before normalize). Getting either of these wrong drops the
  pretrained mAP from ~0.7 to <0.05 on this val set.

### KD as cls-quantization-recovery lever (PTQ collapse partial fix)
- Where: `training/edge/yolo/distill_train.py` + `training/edge/results/US-006.md`
- Pattern: Where US-004's PTQ produced mAP=0 and US-005's QAT couldn't
  recover (per-tensor activation scale dominated by xywh), KD produces a
  student whose cat-channel logits are pre-shaped by the teacher's
  calibrated cls magnitudes. The post-PTQ output tensor's per-tensor scale
  no longer crushes cls into a single int8 bucket. Result: 0.000 → 0.205
  mAP under the IDENTICAL US-004 PTQ pipeline (same calibration set, same
  TFLiteConverter settings). Not a full recovery but the first non-zero
  YOLOv8 PTQ in the PoC; the next move (out of scope here) is splitting
  Detect head into separate cls/xywh output tensors so each gets its own
  per-tensor scale.

### Inject loss patch via on_train_start callback (avoid trainer subclass)
- Where: `training/edge/yolo/distill_train.py:train_distilled`
- Pattern: ultralytics' `model.train()` opens its own trainer wrapping
  `model.model`. To swap `model.model.loss` AFTER the trainer has placed
  it on the right device, register an `on_train_start` callback via
  `model.add_callback('on_train_start', fn)` and do the patch inside the
  callback (where `trainer.model` is the live device-resident copy). Same
  hook avoids subclassing DetectionTrainer or touching ultralytics'
  internals. Used here for KD; reusable for any loss-modification hack.

### YOLOv8 monkey-patch model.loss for KD without double-forward
- Where: `training/edge/yolo/distill_train.py:patch_student_loss_for_distill`
- Pattern: ultralytics' `BaseModel.loss(batch, preds=None)` computes
  `preds = self.forward(batch['img']) if preds is None else preds`. So a
  patched loss that computes student preds ONCE and passes them via
  `original_loss(batch, preds=s_features)` avoids running the student
  forward twice per step. Saves ~33% of training wall-clock vs naively
  re-forwarding inside the patched loss. Capture `original_loss = student.loss`
  BEFORE assigning the new closure; bound-method reference stays valid.

### Detect head channel layout: [4*reg_max box-DFL, nc cls]
- Where: `ultralytics.nn.modules.head.Detect.forward` — referenced in
  `training/edge/yolo/distill_train.py:_slice_cat_logits`
- Pattern: in TRAINING mode, ultralytics' Detect head returns a list of
  per-scale tensors (B, 4*reg_max + nc, H, W). The first 4*reg_max=64
  channels are the box-DFL distribution; remaining nc are cls logits. To
  align cat across architectures: student (single_cls, nc=1) → cls idx 0
  i.e. tensor[:, 64:65, :, :]; teacher (yolov8s, nc=80) → cls idx 15
  (COCO_CAT_CLASS) i.e. tensor[:, 64+15:64+16, :, :]. Both yolov8n and
  yolov8s share strides (8/16/32) so spatial grids match for any common
  imgsz — no resampling needed for elementwise distillation losses.

### prepare_qat asserts training mode — flip before call
- Where: `training/edge/yolo/qat_finetune.py:qat_one_epoch`
- Pattern: `torch.ao.quantization.prepare_qat(module, inplace=True)` raises
  `AssertionError: prepare_qat only works on models in training mode` if
  the module is in eval mode. ultralytics' `YOLO(...)` loads checkpoints
  in eval mode by default. Call `model.model.train()` before
  `prepare_qat(...)`. Without this, the QAT branch silently falls back
  to plain fp32 fine-tune (which still produces an int8 .tflite but
  doesn't actually exercise fake-quant nodes during the epoch).

### YOLOv8 PTQ failure mode survives QAT — it's an activation-quant problem
- Where: documented in `training/edge/results/US-005.md` ("Why QAT didn't help")
- Pattern: QAT inserts fake-quant nodes that simulate weight quantization
  noise during training. It does NOT alter the per-tensor *activation*
  quantization scales chosen by `tf.lite.TFLiteConverter` at export time.
  YOLOv8's monolithic Detect output `(1,5,1029)` mixing xywh and cls into
  one tensor will still pick a single scale dominated by xywh, regardless
  of QAT. The fix requires either (a) Detect-head split into separate
  output tensors (touches ultralytics' export graph; out of scope for
  US-005), (b) per-channel activation quant (not supported by TFLM on
  ESP32-S3), or (c) a smaller student whose head quantizes more cleanly
  (US-006 KD). Mark mAP=0 honestly and move on.

### TF_USE_LEGACY_KERAS=1 is mandatory before onnx2tf
- Where: `training/edge/yolo/export_quantize.py` (top-of-module env var setdefault)
- Pattern: TF 2.17 ships Keras 3 by default; onnx2tf uses Keras-2-style
  symbolic tensor APIs that crash on Keras 3 with "A KerasTensor cannot be
  used as input to a TensorFlow function". Set
  `os.environ.setdefault('TF_USE_LEGACY_KERAS', '1')` BEFORE any TF/Keras
  import (i.e. at module top, before lazy imports get triggered). Required
  by US-004, US-006, US-009 — anything that uses onnx2tf.

### Pre-place onnx2tf's calibration_image_sample_data .npy
- Where: repo root `calibration_image_sample_data_20x128x128x3_float32.npy`
  (gitignored)
- Pattern: onnx2tf tries to download a 20×128×128×3 fp32 .npy at convert
  time; the github-releases URL it uses is stale and the wasabi fallback
  is flaky from many networks. Pre-creating the file in `cwd()` short-
  circuits the download (`os.path.isfile` check at common_functions.py:4271).
  Any random fp32 array of that shape works; we use a seeded RandomState.

### YOLOv8 PTQ + per-tensor output quant collapses cls head
- Where: documented in `training/edge/results/US-004.md` ("Failure mode")
- Pattern: YOLOv8's monolithic Detect output `(1, 5, 1029)` mixes xywh
  (pixel range 0..480) with cls (sigmoid 0..1) into one tensor. Naive PTQ
  picks ONE per-tensor scale dominated by the xywh range, which compresses
  the cls range to a single int8 bucket — every confidence becomes 0,
  every detection is filtered out, mAP=0. Mitigations: (a) QAT (US-005),
  (b) split Detect head into separate tensors (out of scope for US-004
  because it requires touching ultralytics' export graph), (c) KD to a
  smaller student (US-006). The MLIR per-channel quantizer
  (`experimental_new_quantizer=True`) does NOT fix this — per-tensor
  activation quant still wins on the output tensor.

### Decode YOLO ONNX/TFLite output: divide xywh by imgsz
- Where: `training/edge/eval/adapters/onnx_adapter.py:_decode_yolo_onnx`
- Pattern: ultralytics' `boxes.xyxyn` (used by `pytorch_adapter.py`)
  returns NORMALIZED [0,1] coordinates, but the raw ONNX/TFLite output is
  xywh in INPUT-PIXEL coordinates (0..imgsz). Without dividing by imgsz
  before clipping to [0,1], every box collapses to a degenerate point and
  mAP is 0 across all eval harness format paths. The decoder now takes
  imgsz as a third arg; both onnx_adapter and tflite_adapter pass it.

### TFLite "params" reported by the eval harness over-counts

**Learnings**:
- KD is the ONLY recovery lever that worked for YOLOv8 PTQ in this PoC.
  US-004 (PTQ) → 0.000 mAP; US-005 (QAT) → 0.000 mAP; US-006 (KD+PTQ) →
  0.205 mAP. The fp32 weights are pre-shaped by the teacher's cat-channel
  logit magnitudes such that the post-PTQ per-tensor scale on the
  monolithic Detect output no longer crushes cls into one int8 bucket.
  The mechanism is structural — KD doesn't change the architecture; it
  changes the activation distribution that the calibrator sees.
- Distilled fp32 mAP went from US-003's 0.909 to 1.000 — saturated on the
  120-image val. The val is small enough that 1.000 is plausible but not
  meaningfully different from "near-perfect"; cite it as evidence the
  distillation didn't hurt training.
- Channel alignment was the trickiest design decision: student is
  single_cls (nc=1) but teacher is COCO (nc=80). Distilling on softmax
  across classes is degenerate for the student. Settled on per-channel
  binary KD — slice teacher's cat (idx 15) and student's class-0,
  match via temperature-scaled BCE. T**2 scaling is the Hinton
  prescription so the gradient magnitude is invariant to T at small
  logits.
- Patch-via-callback (`on_train_start`) is cleaner than subclassing
  DetectionTrainer. The trainer hands us the device-resident model at
  callback time; that's the right point to swap `model.loss`. The same
  pattern would work for any loss modification (regularization, KD,
  consistency losses).
- Captured a real subtlety: when the trainer calls
  `self.loss, self.loss_items = self.model(batch_dict)`, the
  BaseModel.forward dispatches to self.loss(batch). Our patched loss
  computes student preds ONCE and passes them via the original
  bound-method `original_loss(batch, preds=s_features)`. Without that,
  the student forward runs twice per training step (~33% overhead).
- Out of scope but worth recording for US-012 follow-ups: the remaining
  79.5 mAP-point PTQ gap is structural — Detect head emits cls + xywh
  on a single output tensor sharing one per-tensor scale. Splitting the
  head into separate output tensors (which ultralytics does for its
  built-in tflite export when `int8=True` is passed) is the obvious
  next step. KD + head-split is the combination that the literature
  suggests would close the gap.
- Wall-clock: 50 epochs on 480 train / 120 val @ batch=32, imgsz=224 on
  a single RTX 4090 took ~75s. That's negligibly slower than US-003's
  plain fine-tune (~70s) — the teacher's frozen forward is cheap at
  yolov8s scale, single-forward patching keeps the overhead near zero.

  load-bearing for the firmware C port's correctness, not its budget.
- The IN_GRAPH_NMS_OPS set is conservative — it lists the four op
  names the TFLite schema uses for post-processing
  (TFLite_Detection_PostProcess for SSD, NON_MAX_SUPPRESSION_V4/V5
  for newer exporters, and TF's CombinedNonMaxSuppression which can
  survive into a custom TFLite delegate). Anything else is treated as
  "not NMS". If a future architecture ships post-processing under a
  different op name, add it to this set; the test
  `test_in_graph_nms_ops_set_covers_known_exporter_outputs` is a
  regression bumper for that change.

**Learnings**:
- The same exclusion rule (status==passed AND tflm_compatible AND
  mAP@0.5 > 0) keeps the v2 frontier honest. Five of the eleven iter-*
  candidates (4 + iter-D-blocked-fixture) collapsed to mAP=0 under
  per-tensor activation quant — surfacing those in the matrix with
  pareto_verdict=regress (and excluding them from the frontier list)
  is more useful than silently dropping them. Future iterations
  should keep this contract.
- Frontier sort (fps desc) and winner pick (mAP desc) are two distinct
  rankings. They diverge any time the highest-mAP row isn't the
  fastest. Codifying them as separate functions with explicit docstrings
  makes the SUMMARY's recommendation paragraph reproducible and
  testable.
- iter-G's JSON shape is intentionally different — same .tflite as
  iter-A but a refreshed inference_only_latency. _extract_candidate
  carries that as a third branch (alongside candidate_metrics and bare
  EvalResult). When future iterations introduce new measurement-only
  variants (e.g. fp16 fallback, profile-guided arena tuning), they
  should follow this pattern: emit iter_a_baseline_metrics +
  measurement_only_latency, not a synthetic candidate_metrics block.
- The deprecation list test (test_deprecation_list_includes_v1_INT8_candidates_dominated)
  enforces strict accuracy domination as the v1->v2 displacement rule.
  Latency/size improvements alone don't trigger deprecation because
  v1's frontier is implicitly the v2 floor (iter-A baseline). Codifying
  this avoids accidentally deprecating a slower-but-more-accurate v1
  candidate.
- The recommendation paragraph weaves the four PRD-required clauses
  (PSRAM, MIPI-CSI vs HTTP, classifier handoff, fps caveat) into one
  narrative. A test that asserts each clause's keyword appears in the
  output is a low-cost guardrail against future copy-edits silently
  removing a section.
